About HTML

Hyper Text Markup Language, HTML, is the standardized language for writing web documents. It creates a framework for content to be displayed on browsers. The framework is constructed by a set of tags. Common HTML tags, also called elements, include <span>, <div>, <a>, <img>, etc. A complete list of elements can be found at MDN's "HTML element reference". Learn HTML and CSS with Codecademy's HTML & CSS learning app.

Structure

The basic structure of an HTML document consists of the <!doctype html> declaration tag, the <html> tag and its two direct child tags, <head> and <body>.

<!doctype html>
<html>
    <head></head>
    <body></body>
</html>

The <head> consists of meta data (data that is not shown on screen). This is where the developer inserts titles, CSS file paths, and necessary JS file paths.

<head>
    <meta charset="utf-8">
    <title>Page Title</title>
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <link rel="stylesheet" href="css/css/modern.css">
</head>

The <body> consists of browser visible content. The structure of a basic page body would look something like this:

<body>
    <div class="header"></div>
    <div class="mainContent"></div>
    <div class="footer"></div>
</body>

JS paths are also placed inside the <body>. This method allows the HTML and CSS to be loaded before the JS, improving the user experience. Always place JS paths at the end of the <body>, unless it is necessary for a specific JS file to be loaded before the HTML and CSS. Then place the JS path in the <head>.

<body>
    <div class="header"></div>
    <div class="mainContent"></div>
    <div class="footer"></div>

    <script src="js/vendor/jquery-1.11.3.min.js"></script>
    <script src="js/main.js"></script>
</body>

Object Naming

There are two attributes that can be used for naming elemental tags: the id and the class. The id can only be used once per page. It should only be used for non-reusable items like anchor links. The class attribute can be used multiple times on a page and is the preferred method of object naming. Classes also have a lighter CSS specificity weight than ids, so they are easier to style and extend styles.

Best Practices

  • Use classes for naming objects and use lowercase
  • Use the lower camel case naming convention for compound worded names
  • Child elements should inherit the parent name using the -, aka dash, to chain the parent name to the child name
  • Avoid chaining more than three words together
  • Do not mix object names together in a single tag.
  • Do not intertwine complex objects. Make sure they are properly nested when combining objects together.

<div class="mediaBox"></div>
<ul class="list"><li></li></ul>
<ul class="hList"><li></li></ul>

Aria Roles

The Accessible Rich Internet Applications Suite, ARIA, adds accessibility to websites and applications by assigning 'roles' to tags. These roles help screen readers navigate to specific areas of the page/app. The best practice is to add roles to the general landmark sections. Adding more roles than the landmarks is up to the developer's discresion. "ARIA Roles 101" has the full list of ARIA roles.

Important Landmark Roles

  • banner (apply to header)
  • complementary (apply to sidebars)
  • contentInfo (apply to footer)
  • form (apply to form tag)
  • main (apply to main content)
  • navigation (apply to navigation)
  • search (apply to search form)

Example 

<body>
    <div class="header" role="banner">
        <div class="logo"></div>
        <ul class="nav" role="navigation">
            <li><a href="#">link 1</a></li>
            <li><a href="#">link 2</a></li>
            <li><a href="#">link 3</a></li>
        </ul>
        <form class="search" role="search">
            <label for="globalSearch" class="search-label"><i></i></label>
            <input type="search" name="search" id="globalSearch" class="search-field" placeholder="Search">
            <input type="submit" class="search-submit" />
        </form>
    </div>
    <div class="mainContent" role="main">
        <!-- MAIN CONTENT... -->
        <form class="contact" role="form">
            <!-- CONTACT FORM FIELDS... -->
        </form>
    </div>
    <div class="sideBar" role="complimentary">
        <!-- SIDEBAR CONTENT... -->
    </div>
    <div class="footer" role="contentInfo">
        <!-- FOOTER CONTENT... -->
    </div>
</body>

Validation

It is best practice to validate each page's HTML for errors based on W3C standards. Modern browsers are fairly good at parsing error riddled code, but they still miss some. There is also a strong possibility that the main testing browser parses the error without issue, but a different browser does not. The most common error is caused by the failure to add an end tag. This simple error can break an entire page. It is not always possible to fix all validation errors, as 3rd party code does not always comply. A simple rule of thumb is to make sure that the code written by the developer is validated at 100% while giving 3rd party code (plugins, snippets, links, boilerplate meta data) more leeway.

Reasons for Validating:

  • Validation is a useful debugging tool
  • Future proofing code for the next gen browsers
  • Easier future maintenance and updates
  • Establishes standards and practices in the developer global community
  • Establishes professional code

For more information on Validation, read "Why Validate?" There are three ways to validate: by URL, file upload, and direct input. URL can only be used if the page is hosted publicly and is quick for live site validation. Development testing is best suited for the direct input approach. Start validating your code at W3C's Validator.
