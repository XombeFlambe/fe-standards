Writing CSS

CSS (cascading style sheet) is a language used to style markup (HTML for example). It can be written in .css files, directly in the html element tag, and inside a style tag located in the head tag of an html document. SCSS (sassy css) is an updated version of SASS (semantically awesome style sheets). It is a pre-processing language for CSS code that allows the developer to modularize, create variables, create mixins, nest css code, and use math operators. Think of SCSS as more of an extension of CSS rather than a standalone language. SCSS is the Wiley standard for writing CSS and should be used for all current and future projects. New to CSS? Codecademy has a great set of learning modules.

CSS

  • It is best practice to write your CSS code in .css and .scss files. This allows CSS to be reuseable for many shared objects on many pages.
  • Avoid writing CSS code in the markup (also called inline styles), as the code is not reuseable and will overwrite styles in .css files. When choosing to write CSS in the markup, make sure there is a valid reason for doing so.
  • Use the class attribute as the selector whenever possible to name your element tags for styling purposes. Class attributes can be used multiple times on a page.
  • Only use the id attribute as the selector for unique objects, as id's can be used once on a page. Using an id more than once per page will break the code. An example of proper id use is for hyperlinking. The sections of this page are hyperlinked to their respective sub navigation links using unique id names.

SCSS

  • Read "Sass Basics" to learn how SCSS works.
  • A CSS compiler is needed to process .scss files into .css files. Prepos is a standalone compiling app that has been used for the Everything DiSC and Five Behaviors Kentico sites. If the project uses a task runner, then a compiler can easily be added to the task file instead of using a standalone app like Prepos.
  • Avoid using the @extend feature. Instead, write extender classes.
  • Limit the use of nesting css to keep objects modular.

CSS Anatomy

The anatomy of a CSS rule consists of a selector and a declaration. The declaration consists of of a property and a value. Read "Anatomy of CSS Rules - Selectors & Declarations" for more documentation. Selectors can be comprised of classes, ids, attributes, elements, pseudo classes, and psuedo elements. "The 30 CSS Selectors you Must Memorize" is a must read resource for learning how to effectively create selectors. W3C has a full property table.

Anatomy

selector {
    property: value;
    property: value;
    property: value;
}

Heirarchy

Stack the properties in specific order and groupings as displayed below:

selector {
   display (display)
   box-model (width, height, padding, margin, border, etc.)
   positioning (relative, absolute, top, left, bottom, right, z-index, etc.)
   color/background (color, background, background-positioning, background-image, etc.)
   fonts (font-family, font-size, font-weight, etc.)
   text (text-align, vertical-align, line-height, letter-spacing, etc.)
   misc (cursor, etc.)
   transformations (transform, transition, animation, etc.)
}

Example

.apple {
    display: block;
    width: 50px;
    margin: 20px;
    border: 2px solid purple;
    position: absolute;
    top: 20px;
    left: 50%;
    color: red;
    font-size: 16px;
    line-height: 1.4;
    cursor: pointer;
    transform: rotate(30deg);
}

CSS Specificity

CSS specificity determines how the browser decides what styles take precedent over other styles using a weight system. There are many ways to style elements, and they each have specific weights.

  • The !important css rule overrides all weights.
  • Direct inline styling has a weight of 1000 and will override any object less than 1000
  • ids have a weight of 0100 and will override any object with a weight less than 0100
  • classes, pseudo-classes and attributes have a weight of 0010 and will override any object less than 0010
  • elements and pseudo-elements have a weight of 0001

It is best practice to style objects with lighter weights. Weight specificity can be a blessing like using mixin classes that make objects more customizable. It can also be a curse by unintentionally overwriting styles. Read "Specifics on CSS Specificity" to learn how to weigh specificity. Hone your weight specifity skills with interactive flash cards.

CSS Resets

All browsers have default styles on html elements, and some of these styles differ between Chrome, IE, Firefox and the various other browsers. This inconsistency of styles can make your layouts look great on one browser and break on another. IE has had many default styling issues across all their browser versions, and they currently continue to add more default styles than Chrome or Mozilla. In Early 2014, Firefox decided to update and add to their default styling, but this decision created even more inconsistency. Reset styles are used to zero out the default browser styles that cause inconsistencies. These styles must be placed at the top of your CSS file so that you can override them with your own custom styles. You can google css reset generators (for example) online or use HTML5 Boilerplate that includes the reset styles. Below is an example of the reset styles used for this project.

Display Property

The display property is used to change the normal display of an element. A span element can be changed into a block object or a div element into an inline object. There is also a hybrid between the two, inline-block, which keeps the object inline but can also take on margin properties. Margins cannot be used on inline objects.

Example

NOTE: The style properties are placed inline and/or inside a style tag in this example for display purposes only.

<div style="padding:5px; margin-bottom: 20px; display:inline-block; background:gray;">Div element, display inline-block</div>

Floats

Floats are a means of positioning objects to the left or right of the parent element, and they can be used on block objects to mimic inline objects that stack next to eachother instead of on top of eachother. Grid and block list objects are a perfect example of floating divs. Always remember to clear your floated objects or they may not display properly. Add a clear fix to the parent element of the floated object to achieve this.

Example

Browser View

HTML

<ul class="blocks blocks_4col">
    <li><img src="http://placehold.it/400x200" class="imgResponsive" alt="Block example"></li>
    <li><img src="http://placehold.it/400x200" class="imgResponsive" alt="Block example"></li>
    <li><img src="http://placehold.it/400x200" class="imgResponsive" alt="Block example"></li>
    <li><img src="http://placehold.it/400x200" class="imgResponsive" alt="Block example"></li>
</ul>

CSS

.blocks {margin-left: -20px;}

.blocks:before,
.blocks:after {
    display: table;
    content: "";
}

.blocks:after {clear: both;}

.blocks > li {
    float: left;
    margin: 0 0 20px 20px;
}

SCSS

/* Located in the scss/helpers/_utils.scss file */
@mixin clearfix() {
    &:before,
    &:after {
        content: " ";
        display: table;
    }

    &:after {
        clear: both;
    }
}

/* Located in the scss/modules/_blocks_md.scss file */
.blocks {
    margin-left: -$BLOCK_MARGIN;
    @include clearfix();
}

.blocks > li {
    float: left;
    margin: 0 0 $BLOCK_MARGIN $BLOCK_MARGIN;
}

Z-Index Property

Z-indexes specify the 'stacking' order of an object on the z-axis. Avoid setting large z-indexes, as many developers fall victim to index creep, for example: A developer may have a tooltip z-indexed at 9999. A client then wants to include a sticky header, so the developer then adds a z-index of 99999 to the header. Next the client wants to include a modal, so the developer adds a z-index of 999999. This happens more often than not and is bad practice. This project's modal uses a z-index of 9 to achieve it's top level heirarchy. Keep your z-indexes maintainable using low numbers. There is no need to use anything larger than 10 to achieve proper z-index heirarchy. There is much more to z-indexing than just making a number bigger to make an object's heirarchy better. Read "What No One Told You About Z-Index" for more info on how z-indexing works.

!important

!important is a css rule that can be added to a selector property's value and tells the browser to ignore all other values of that specific selector property. It is best practice not to use !important unless there is a valid reason and strong argument to use it. Casual use of !important can and will cause future css maintenance issues.

Valid reasons for using !important:

  • Overriding 3rd party plugin styles. 3rd party plugins include styles that you will want to customize without editing the plugins CSS file. Plugin styling can be over complicated and sub-standard. Override the plugin styles in your own CSS files, and if need be, use the !important rule.
  • Working with unmanageable specificity weights. When a project suffers from code bloat after multiple maintenance fixes, the css selector chains can become so long that it is no longer in the best interests of the developer to bother with specificity weights.
  • Working on old sites using inline styling. The inline styling is sub-standard and out-dated, and the !important can save time and headaches dealing with it.

Example

SCSS overrides for the 3rd party plugin, flexslider, used in this project.

.flexslider {
    margin-bottom: 0 !important;
    border: none !important;
}

.flex-control-nav {
    bottom: 20px !important;
    z-index: 2 !important;
}

.flex-control-nav {bottom: -10px !important;}

.flex-direction-nav {display: none;}

.flex-direction-nav a:before {color: $BRAND_LIGHT !important;}
.flex-control-paging li a {border-color: $BRAND_LIGHT !important;}
.flex-control-paging li a:hover,
.flex-control-paging li a.flex-active {background: $BRAND_LIGHT !important;}

.flexslider img {margin-bottom: 20px !important;}

.slides-info {
    width: 100%;
    text-align: center;
}

.slides-info-hd {
    margin-bottom: 20px;
    padding: 0 10px;
    font-size: 28px;
}

Extender Classes

Extender classes are used to expand the styling of their parent class. They help keep CSS modular and reuseable and are used on a number of objects in this project. There are a number of objects in this project that use extender classes. Take a look at how buttons are styled in the example below.

Example

The base class is .btn, and it has to extenders. One extender, .btn_white overrides the base class background, color and border property values. The other extender, .btn_sm, overrides the base class min-width, padding and font size property values. Multiple extenders can be used on a single base class to create even more variations, like the white small button in this example.

<button href="#" class="btn btn_secondary">Secondary Button</button>

Mixin Classes

Mixin classes are similar to extender classes, but they do not have parent classes. Instead, they are used globally on many objects and are an exeption to the rule of not mixing selectors. Some are directly styled, and some have no style properties but are indirectly styled when chained with another class. Mixin classes should be used to enable/disable or show/hide objects. Avoid creating mixins when an extender class is more appropriate, like creating a white version of the blue button.

Example Using Styled Mixin Class

There are two buttons in this example, but the second button contains the .isVisuallyHidden class, which visually hides the button from view but is still visible in the browser inspector.

Common Mixins used for Chaining Classes

  • .isHidden
  • .isVisuallyHidden
  • .isHiddenMobile
  • .isHiddenDesktop

Example Using Unstyled Mixin Class

Text Editor (WYSIWYG) Styles

Wysiwyg (what you see is what you get) is an industry standard acronym for a CMS (content management system) text editor. The wysiwyg text editor is used by a CMS administrator to make content changes without having to be versed in html code. Wysiwyg styling is achieved by directly styling html elements and using a parent .wysiwyg container div. It is best practice to create a wysiwyg module when using SCSS, instead of creating a separate style sheet.

Example

The most common elements wysiwyg text editors used are <b>, <i>, <u>, <p>, <a>, <h1>-<h6>, <ul>, and <ol>. These elements must be contained by a .wysiwyg parent div. The <p> and <a> elements will inherit your already created styles, but if you wish to make the wysiwyg styling for these elements different, you can create the new styling within the .wysiwyg parent class. Text editor tools work strictly with element tags. Adding classes to these tool tasks are rarely an option and would require customizing back end code.

SCSS

SCSS allows you to nest classes and elements. Nest your elements inside the .wysiwyg class. If you are using straight CSS, then you will need to place .wysiwyg ahead of each element tag (for example, .wysiwyg p {margin-bottom: 10px;}).

.wysiwyg {
    h1, h2, h3, h4, h5, h6 {
        margin-bottom: 10px;
        font-weight: bold;
    }

    h1 {font-size: 40px;}
    h2 {font-size: 32px;}
    h3 {font-size: 24px;}
    h4, h5, h6 {font-size: 18px;}

    p {margin-bottom: 20px;}

    b {font-weight: bold;}

    i {font-style: italic;}

    u {text-decoration: underline;}

    ul, ul li {list-style-type: disc;}

    ol, ol li {list-style-type: decimal;}

    ul, ol {margin: 0 0 20px 20px;}
}
